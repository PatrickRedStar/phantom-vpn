# v0.26.21 — Three-bug fix release (Android)

**Дата:** 2026-06-05
**Версия:** 0.26.21 (versionCode 95)
**Релиз-тип:** patch — корректность диагностики + телеметрии туннеля.

## Контекст

При проверке свежеустановленного APK `io.ghoststream.vpn.debug` (versionCode 94, 0.26.20-debug)
на устройстве Samsung Tab S11 обнаружено три связанных бага:

1. **Phantom tunnel** — через ~2 минуты после Connect интернет фактически пропадает,
   но UI рапортует `state:"connected", health:"healthy", last_error:null`.
2. **Кнопка "Поделиться" в логах не работает** — `Failed to find configured root that
   contains /data/data/io.ghoststream.vpn.debug/cache/ghoststream-session.log`.
3. **Отладочный отчёт не включает логи** — пользователь делает Settings → Diagnostic →
   Share, в текстовом отчёте секция `--- Логи ---` пуста или содержит «Логи пусты».

## Bug 2 — FileProvider crash в Logs → Share

### Root cause

`LogsViewModel.shareLogs:261` пишет файл прямо в `cacheDir/`:

```kotlin
val out = File(context.cacheDir.apply { mkdirs() }, "ghoststream-session.log")
```

`file_paths.xml` декларирует только подпапку `logs/`:

```xml
<paths>
    <cache-path name="logs" path="logs/" />
    <cache-path name="debug" path="debug/" />
</paths>
```

`FileProvider.getUriForFile` рейзит `IllegalArgumentException: Failed to find
configured root` — корень `cacheDir` не объявлен. Exception ловится внешним
`runCatching` в shareLogs, пользователь видит Toast "Не удалось: …".

### Исправление

Один файл, одна строка — `LogsViewModel.shareLogs:261`:

```kotlin
// до:
val out = File(context.cacheDir.apply { mkdirs() }, "ghoststream-session.log")
// после:
val out = File(File(context.cacheDir, "logs").apply { mkdirs() }, "ghoststream-session.log")
```

Файл попадает в `cacheDir/logs/`, который покрыт `<cache-path name="logs" path="logs/" />`.

`paths.xml` не трогаем — оставить explicit подпапку лучше чем root `<cache-path
path="" />`, чтобы случайные файлы в `cacheDir` не уходили в Intent наружу.

## Bug 3 — Debug report silent failure

### Root cause (предполагаемый)

`SettingsViewModel.shareDebugReport:512` не обёрнут в `runCatching`. Если на каком-то
этапе вылетает Exception (FileProvider, IO, или новый — пока неизвестный), он
поглощается `viewModelScope` без Toast'а пользователю. UI просто не реагирует.

Альтернатива: текст отчёта успешно открывается в Share Sheet, но секция
`--- Логи ---` пуста, потому что `LogPersister.tailLines(2000)` вернул пустой
список — например, из-за гонки между `LogPersister.start()` (вызывается в
`MyApplication.onCreate`) и кликом на Share. На устройстве файл `ghoststream.log`
размером 28KB, значит `LogPersister` работает; но `tailLines` может вернуть пусто
на короткий момент после flush.

### Исправление

В `SettingsViewModel.shareDebugReport:512`:

1. Завернуть всё тело coroutine в `runCatching { … }.onFailure { e -> Toast +
   Log.e }` — пользователь увидит причину сбоя.
2. Добавить диагностический маркер сразу перед секцией логов:

   ```kotlin
   val logFile = java.io.File(context.filesDir, "logs/ghoststream.log")
   val persistBytes = if (logFile.exists()) logFile.length() else -1L
   sb.appendLine("--- Логи (persist=${persistBytes}B, tail=${lines.size} строк) ---")
   ```

   Маркер `persist=N, tail=K` диагностический: если `N>0, K=0` — баг в
   `tailLines`, добъём отдельно. Если `N=0` — `LogPersister` не пишет, копаем
   producer (VpnStateManager.logFrames).

3. Если в новом отчёте видно `persist>0, tail=0` — следующий патч уточнит
   `tailLines` (например, добавит `readLines()` fallback на повреждённый файл).

## Bug 1 — Phantom tunnel в supervise.rs

### Root cause

Поведение из лога устройства:
- Через ~2 минуты `[tls_rx_loop]` любого stream'а не получает RX 45 сек.
- `tls_tunnel.rs:71-83` зашит на `Err("rx idle timeout: no data for 45s")` и
  завершается. Stream task в `supervise.rs:754-790` ловит Err, проставляет
  `tele.streams_alive[idx]=false`, пишет WARN `[stream] kill err_kind=timeout`.
- **Supervisor не реагирует** на одиночные kills — `drive_tunnel` ждёт глобального
  shutdown-сигнала (`forwarder_dead_rx`, cancel, или *всех* tasks завершившихся).
- `streams_alive` derived в `streams_up` (status frame). Пока хотя бы один stream
  жив, `drive_tunnel` крутится. `f.health` остаётся `Healthy` (выставлен на
  Connect-trans `supervise.rs:593`), `f.last_error` остаётся `None`.

Итог: 5 из 8 streams умерли, эффективная пропускная способность ~0 байт/с (хеш
по 5-tuple шлёт пакеты на убитые streams), UI говорит `healthy`. Reconnect
не запускается.

### Исправление

#### 1. Death watcher в `drive_tunnel`

Новый async task рядом с `telem_task` (~`supervise.rs:795`). Каждую секунду
читает `streams_alive` через atomic snapshot:

```rust
let dead_watcher = {
    let tele = telemetry.clone();
    let status_tx = status_tx.clone();
    let forwarder_dead_tx_clone = /* clone or restructure */;
    tokio::spawn(async move {
        let mut dead_streak_secs = 0u32;
        loop {
            tokio::time::sleep(std::time::Duration::from_secs(1)).await;
            let alive = (0..n_streams)
                .filter(|i| tele.streams_alive[*i].load(Ordering::Relaxed))
                .count();

            // streams_up == 0 ≥ 3 s → teardown.
            if alive == 0 {
                dead_streak_secs += 1;
                if dead_streak_secs >= 3 {
                    let mut f = status_tx.borrow().clone();
                    f.last_error = Some("all streams dead".to_string());
                    f.health = TunnelHealth::Dead;  // или существующий вариант
                    let _ = status_tx.send(f);
                    let _ = forwarder_dead_tx_clone.send(());
                    return;
                }
            } else {
                dead_streak_secs = 0;
            }

            // streams_up < n_streams/2 → degraded (не teardown).
            let half = (n_streams + 1) / 2;
            if alive < half {
                let mut f = status_tx.borrow().clone();
                f.health = TunnelHealth::Degraded;
                f.last_error = Some(format!("degraded: {}/{} streams", alive, n_streams));
                let _ = status_tx.send(f);
            }
        }
    })
};
```

Нюанс: `forwarder_dead_tx` — `tokio::sync::oneshot` (supervise.rs:666), уже передан
во владение `rx_forwarder` (стр. 692). Watcher не может слать в тот же канал.
План реализации заведёт отдельный `oneshot` `dead_watcher_tx`/`_rx` и
добавит вторую `tokio::select!`-arm в `drive_tunnel` (как существующая на стр. 884
`_ = &mut forwarder_dead_rx => { … }`).

`TunnelHealth` сейчас содержит `Healthy / Stale / Degraded` (`crates/gui-ipc/src/lib.rs:337`).
Расширяем `Dead` — Kotlin/Swift биндинги десериализуют enum по строке, новые значения
обратно-совместимы (если UI не знает `Dead`, упадёт в default-arm). Spec плана
включает обновление Kotlin `StatusFrame.health` decoding и iOS аналогичного.

#### 2. TCP keepalive логирование

`tls_handshake.rs:110`:

```rust
// до:
let _ = SockRef::from(&tcp).set_tcp_keepalive(&keepalive);
// после:
if let Err(e) = SockRef::from(&tcp).set_tcp_keepalive(&keepalive) {
    tracing::warn!(
        category = "handshake",
        event = "tcp_keepalive_failed",
        error = %e,
        "set_tcp_keepalive failed — NAT timeout may strike"
    );
}
```

Если на Android видим `tcp_keepalive_failed` — root cause phantom tunnel'а
(keepalive не применён), patch-2 будет ставить keepalive до VpnService.protect()
или через другой механизм.

#### 3. Respawn убитых streams — НЕ в этом релизе

Reupload `tls_writer`/`tls_reader` в живой `drive_tunnel` требует переделки
ownership Vec'ов (`tls_writers`/`tls_readers`) и hand-off задач без teardown.
Объём: 100+ строк, риск регрессии. Откладываем до v0.27.x с отдельным дизайном.

### Что меняется в API телеметрии

- `TunnelHealth` enum может расшириться (`Dead`) — проверить в `gui-ipc`.
- `last_error` теперь может содержать "degraded: X/Y streams" и "all streams
  dead" — UI просто отображает строку, breaking change нет.

## Risk

- **Watcher false-positive:** `streams_alive[i]` атомик. Между snapshot'ом
  watcher'а и стартом нового stream может быть короткое окно где `alive=0`.
  Mitigation: 3-секундный streak до teardown.
- **TunnelHealth::Dead variant:** добавление в `crates/gui-ipc/src/lib.rs:337`
  затрагивает Android Kotlin (StatusFrame deserialization) и iOS Swift.
  План должен включить обновление обоих binding sites.
- **Watcher и dispatcher tx-pipeline:** если все streams умерли, dispatcher
  (`supervise.rs:617`) продолжит читать tun_pkt_rx и `send().await` на мёртвый
  channel вернёт Err — он сам выйдет. Это уже корректно обработано.

## Out of scope

- Stream respawn (откладывается до v0.27.x).
- Изменение `RX_IDLE_TIMEOUT_SECS` (45с — отдельный tuning).
- HTTP/2 server-side keepalive (нужно копать `phantom-server`).
- TLS-layer heartbeat — уже есть, но видно недостаточен в реальных мобильных
  условиях; tuning отдельно.
