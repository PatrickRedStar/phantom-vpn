# Мульти-агентный workflow

Как правильно декомпозировать крупные задачи через параллельных субагентов (`Agent` tool). Инструкции для Claude Code.

---

## Когда использовать

**Задачи, затрагивающие >5 файлов ИЛИ ≥3 независимых файловых зон** — обязательно запускать параллельных субагентов через `Agent` tool одним сообщением. Не делать всё инлайн в главной сессии: это медленно, разбазаривает контекст и увеличивает шанс ошибки.

**Когда НЕ нужен субагент:**
- Точечная правка (1 файл, <50 строк).
- Простой вопрос/ответ без редактирования.
- Задача одной зоны (весь diff в одном crate'е).

---

## Декомпозиция по зонам

Агенты разделяются по файловым зонам с минимальным перекрытием. Типичный mapping:

| Агент | Зона |
|-------|------|
| **Dev-Server** | `crates/server/` |
| **Dev-Android** | `apps/android/`, `crates/client-android/` |
| **Dev-Linux** | `crates/client-linux/`, `apps/linux/` |
| **Dev-macOS-GUI** | `apps/ios/` (UI/Swift) |
| **general-purpose (ios-core)** | `crates/client-apple/` |
| **general-purpose** | `crates/core/`, `crates/client-common/`, `crates/client-core-runtime/` (можно разбить по желанию) |
| **general-purpose (docs)** | `*.md`, `docs/knowledge/`, `server/config/`, `server/scripts/` |

Agent-типы и их специализации — внутренняя классификация Claude Code. `general-purpose` — универсальный, остальные имеют свою «память» и настройки под зону.

---

## Правила

1. **Запуск одним tool-call** — один tool-call с несколькими `Agent` блоками = параллельное выполнение. Последовательные вызовы = серия, медленнее в N раз. Это основная причина держать план в голове и запускать всех агентов сразу.
2. **Non-overlap edits** — агенты не должны править одни и те же файлы. Иначе merge conflicts прямо в рабочем дереве (нет git branch у агентов — всё в одном worktree).
3. **Self-contained prompts** — у агента нет контекста беседы. Каждому давать: цель, затрагиваемые файлы, что уже сделано в других зонах, что именно сделать, как валидировать.
4. **Валидация после merge** — главный агент сам прогоняет `cargo check -p <crate>`, `cargo test`, `./gradlew assembleDebug` и фиксит cross-crate compile errors. Субагенты их обычно пропускают, потому что компилируют только свой crate.
5. **Промпт ≤200 слов на агента** — если не помещается, задача слишком большая, нужно разбить ещё.

---

## Sign of task fit

Задача подходит под мульти-агентный workflow, когда:

- Затронуто **≥3 crate'ов/модулей**.
- **Низкая взаимозависимость** правок между зонами (например, Rust FFI сигнатура стабильна, меняется только одна сторона).
- Можно сформулировать задачу **≤200 слов на агента** — значит зона локальная и понятная.

Если взаимозависимость высокая (сигнатура FFI меняется, все клиенты переезжают одновременно) — лучше сделать пилотную зону инлайн, а параллельно раскатывать только когда контракт фиксирован.

## Пример из истории

**v0.19.4 — удаление QUIC + 4 bug fixes + renaming QUIC_* констант:**
- 4 агента × ~20 минут параллельно вместо ~80 минут инлайн.
- Зоны: server, android, linux-cli, docs. Ни один не трогал чужие файлы.
- Главный агент склеил результат + починил один cross-crate compile error (импорт, про который ни один субагент не знал).

Коммит — `04913e9`. Полный контекст релиза — в [history/timeline.md](history/timeline.md).

## Типичные ошибки

- **Запуск в 4 сообщения вместо одного** — агенты работают последовательно, выигрыш в скорости = 0.
- **Агент А правит `crates/core/src/wire.rs`, агент Б тоже** — merge conflict, один из diff'ов теряется.
- **Забыл дать агенту контекст уже сделанных изменений** — агент не знает, что переменную переименовали, и оставляет старое имя.
- **Главный не проверил `cargo check` после merge** — CI ломается на PR-е, потому что локально никто не пытался собрать всё целиком.
