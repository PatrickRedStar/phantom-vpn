2026-04-11: пользователь попытался добавить второй профиль «hostkey» поверх существующего «vdsina» через UI — зашёл в настройки профиля и вручную поменял `serverAddr` на `hostkey.nl2.bikini-bottom.com:443` (неверный домен — правильный `hostkey.bikini-bottom.com`). Получил DNS error, не смог подключиться.

Цитата: «с этим конечно надо что-то делать, это вообще не USER-FRIENDLY хрень — брать и адрес менять в настройках вручную прописывать его. Хочется просто 2 конфига иметь отдельных».

**Желаемый UX** (варианты):
1. **Кнопка «Клонировать профиль»** — копирует cert/key/admin, спрашивает новое имя и новый serverAddr. Самый простой вариант.
2. **Один профиль с несколькими endpoints** — `serverAddrs: ["tls.nl2...:443","hostkey...:443"]` + switcher на Dashboard. Компактнее в UI, но ломает текущий модель «один профиль = один активный».
3. **Предопределённые пресеты в conn_string** — админ API выдаёт `endpoints: [{name:"direct",addr:"..."}, {name:"relay",addr:"..."}]`, клиент парсит в несколько профилей автоматически при импорте. Требует изменения wire-формата conn_string.

**Why:** Пользователь хочет тестировать оба пути (direct vs relay) и переключаться между ними одним тапом, а не через ручную правку конфига. Ошибка в доменном имени показала, что UI разрешает опасное редактирование, хотя вся остальная установка профиля происходит автоматически через QR/инжект.

**How to apply:** При работе над Android UI (особенно SettingsScreen / ProfilesStore) — приоритетная задача добавить «Clone profile». Минимум: кнопка в меню профиля → диалог с name+serverAddr → сохранить как новый UUID с теми же cert/key/admin. Всё остальное (SNI, tunAddr, adminUrl/Token) наследовать от родителя.

**Временный workaround** (пока не починено): при инжекте через adb на телефон пользователя — сразу делать **два** профиля в `profiles.json` (direct+relay), оба на один cert/key, чтобы пользователь не пытался их создавать в UI.
